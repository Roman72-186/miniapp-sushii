# Отмена автосписания ломалась из-за проверки `contact_id` во фронте

## Симптом

Пользователь нажимает «Отмена автосписания» в настройках, но ничего не происходит. Жалоба выглядит как «кнопка неактивна» или «автосписание не отменяется».

## Причина

Во фронте, в `src/SettingsPage.js`, обработчик отмены начинался с:

```js
if (!profile?.contact_id) return;
```

Но backend `api/cancel-subscription.js` использует только `telegram_id`. Поле `contact_id` приходит в body, но не участвует в проверке и не нужно для отмены.

Из-за этого:
- у пользователей без `contact_id`
- в том числе у части веб-пользователей с ID вида `web_*`

клик по кнопке завершался до `fetch('/api/cancel-subscription')`.

## Как проверить

1. Открыть `src/SettingsPage.js`
2. Найти `handleCancel()`
3. Если там есть ранний выход по `!profile?.contact_id`, баг воспроизводим
4. Открыть `api/cancel-subscription.js` и убедиться, что endpoint требует только `telegram_id`

## Фикс

Использовать валидацию по `telegramId`, а не по `contact_id`:

```js
if (!telegramId) return;
```

Дополнительно стоит исправить текст успешной отмены: backend выключает только следующее списание (`payment_method_id = NULL`), а не деактивирует подписку немедленно.

## Профилактика

- не завязывать рекуррентные платёжные сценарии на `contact_id`
- считать `telegram_id` строковым ключом пользователя, а не обязательно числовым Telegram chat id
- помнить, что `web_*` валиден для отмены и автосписания, даже если Telegram-уведомления для такого ID не доставляются
