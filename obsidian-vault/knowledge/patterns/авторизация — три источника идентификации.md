# Авторизация — три источника идентификации

В приложении три способа понять, кто пользователь. Приоритет — сверху вниз.

## Порядок приоритета

```
1. JWT в localStorage['web_token']
   └─ веб-вход по телефону/паролю
   └─ userId = web_TIMESTAMP_random
   └─ isWebUser = true

2. Telegram WebApp SDK
   └─ window.Telegram.WebApp.initDataUnsafe.user.id
   └─ userId = числовой ID Telegram

3. ?telegram_id= в URL
   └─ fallback для бота (ссылки из уведомлений)
   └─ userId = числовой ID Telegram
```

Реализация: `src/UserContext.js`

## Зачем три источника

- **JWT** — для пользователей без Telegram (заходят через браузер на sushi-house-39.ru)
- **SDK** — стандартный путь для Telegram Mini App
- **URL-параметр** — для ссылок из бот-уведомлений: `/gift-rolls?telegram_id=123`

## Что определяется на основе источника

| Флаг | Откуда |
|------|--------|
| `isWebUser = true` | JWT-токен |
| `isWebUser = false` | Telegram SDK или URL-параметр |

`isWebUser` влияет на:
- Показ кнопок "Войти через Telegram" vs "Выйти"
- Некоторые ограничения функционала

## Идентификаторы

| Тип пользователя | Формат ID | Пример |
|-----------------|-----------|--------|
| Telegram | Числовой | `987654321` |
| Веб-пользователь | `web_` + timestamp + random | `web_1714041234_abc7d` |

## На сервере

Middleware `authMiddleware()` из `api/_lib/auth.js`:
- Проверяет JWT если есть заголовок `Authorization: Bearer ...`
- Устанавливает `req.userId`, `req.userEmail`, `req.authMethod`

Большинство эндпоинтов принимают `telegram_id` из тела запроса напрямую (без JWT).
