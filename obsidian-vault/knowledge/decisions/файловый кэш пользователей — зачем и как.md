# Файловый кэш пользователей — зачем и как

## Что это

`data/users/{telegram_id}.json` — кэш данных пользователя.
TTL: 5 минут. При `force=true` перезаписывается немедленно.

Утилита: `api/_lib/user-cache.js` → `readUserCache()`, `writeUserCache()`

## Зачем

Исторически сложилось при переходе с WATBOT.
WATBOT хранил данные в своём формате — кэш-файлы сохранили ту же структуру.

Текущая роль: снизить нагрузку на SQLite при частых sync-запросах.

## Формат файла (устаревшие русские ключи!)

```json
{
  "variables": {
    "статусСписания": "активно",
    "датаНачала": "01.04.2026",
    "датаОКОНЧАНИЯ": "30.04.2026",
    "PaymentID": "2e5..."
  },
  ...
}
```

> ⚠️ Ключи с **русскими названиями** — унаследованный формат от WATBOT. Конвертация SQLite → кэш выполняется в `deriveFromDbUser()` (`api/_lib/subscription-state.js`).

## Поток обновления

```
POST /api/sync-user
    ↓
Читает SQLite → deriveFromDbUser()
    ↓
Пишет в data/users/{id}.json
    ↓
UserContext.sync() обновляет стейт
```

## Инвалидация кэша

Кэш сбрасывается при:
- `force=true` в запросе sync-user
- Изменении тегов через `POST /api/admin/user-tags`
- После успешного платежа (YooKassa webhook вызывает sync)
