# SQLite как основное хранилище

## Решение

Используется SQLite через `better-sqlite3`. Один файл `data/sushii.db` на VPS.

## Почему SQLite, а не PostgreSQL

- Простота деплоя: нет отдельного сервиса
- Один файл = простой бэкап (`npm run backup:db`)
- Синхронный `better-sqlite3` = нет async/await в БД-коде
- Нагрузка небольшая (мини-апп одного ресторана)

## Ограничения

- Один writer: нельзя масштабировать горизонтально (не актуально)
- Нет нативного JSON-типа (храним как TEXT)
- Даты хранятся как строки DD.MM.YYYY — **не ISO** (историческое решение)

## WAL mode

```js
_db.pragma('journal_mode = WAL');
```

Позволяет параллельные reads при одном writer. Важно для Express, который обрабатывает запросы конкурентно.

## История: удаление WATBOT

До 2026-03-16 основным хранилищем был WATBOT (внешний сервис).
После миграции — SQLite стал единственным источником истины.

Поле `watbot_contact_id` в таблице `users` осталось как артефакт.
