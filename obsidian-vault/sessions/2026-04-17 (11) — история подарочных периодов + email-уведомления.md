# 2026-04-17 (11) — История подарочных периодов в ЛК + email-уведомления

## Что сделано

### 1. История подарочных периодов в /profile
Новая секция «🎁 Подарочные периоды» в аккордеоне `ProfilePage` после «История заказов». Показывает окна согласно тарифу:
- **Claimed** ✓ (прошлые) — дата получения
- **Available** 🎁 (текущее) — осталось дней до конца окна
- **Upcoming** ⏳ (будущие) — откроется через N дней
- **Admin-grants** 👑 — отдельной секцией

Матрица тарифов: **490 → только роллы (15 дней)**, **1190 → только сеты (30 дней)**, **290/без подписки** → блок апгрейда.

### 2. Email-уведомления (Resend.com)
- **«Подарок доступен»** — при открытии нового окна (триггер в `api/get-gift-windows.js`, fire-and-forget)
- **«Автосписание завтра»** — за 1 день до `subscription_end`, только если `payment_method_id` (в `api/cron-subscriptions.js`)
- Идемпотентность через новую таблицу `email_notifications_log` с UNIQUE-ключом `(user_id, notification_type, context_key)`

### 3. Сбор email
- Новая колонка `users.email` (PG + SQLite).
- Email сохраняется автоматически после успешной OTP-верификации (`_otp-store.js` хранит email, `verify-otp.js` вызывает `setUserEmail`).
- При входе в `/profile` без email — обязательная модалка `EditProfileModal` с `requireEmail=true` и пояснением «для уведомлений о подписке, подарках и важных новостях».
- Редактирование email в обычной модалке профиля (опциональное поле).
- Админ может редактировать email через `api/admin-update-user.js`.

## Схема БД (применено на VPS через runtime-миграцию)

```sql
ALTER TABLE users ADD COLUMN email TEXT;

CREATE TABLE email_notifications_log (
  id SERIAL PRIMARY KEY,
  user_id TEXT NOT NULL,
  notification_type TEXT NOT NULL,   -- 'gift_available' | 'renewal_reminder'
  context_key TEXT NOT NULL,          -- 'roll_DD.MM.YYYY' | 'set_DD.MM.YYYY' | subscription_end
  sent_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(user_id, notification_type, context_key)
);
```

Проверено на VPS: колонка `users.email` и таблица `email_notifications_log` созданы (PostgreSQL/Supabase).

## Edge cases покрыты
- Юзер без email → модалка требует заполнить (без чекбокса ПД, т.к. согласие уже дано)
- Юзер без `payment_method_id` → email о продлении НЕ шлём (не вводить в заблуждение — автосписание не будет)
- Тариф 290/без подписки → блок апгрейда вместо окон
- Тариф 9990 (Амбассадор) → автоматических окон нет, только admin-grants
- Идемпотентность cron → UNIQUE в `email_notifications_log`
- Rate limits Resend → `setTimeout(200ms)` между письмами в cron

## Решения
- **Email не сохранялся раньше** — использовался разово для OTP и забывался. Теперь сохраняется в `users.email` автоматически.
- **Обязательный email для старых юзеров** — без чекбокса согласия, т.к. при регистрации они уже дали согласие на ПД (решение пользователя).
- **UI размещение** — в аккордеоне рядом с «История заказов», не в hero (иначе hero раздуется).
- **Админ-grants не шлют email** — администратор сам уведомляет юзера о выданном подарке.
