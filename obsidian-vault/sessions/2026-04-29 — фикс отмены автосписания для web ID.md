# 2026-04-29 — фикс отмены автосписания для web ID

## Что сделано

Исправлен баг, из-за которого часть пользователей не могла отменить автосписание из настроек. Проблема была не в YooKassa и не в cron, а во фронтовой проверке перед отправкой запроса.

## Причина

В `src/SettingsPage.js` отмена автосписания блокировалась условием `if (!profile?.contact_id) return;`.

Это оказалось неверным допущением:
- backend `api/cancel-subscription.js` требует только `telegram_id`
- `contact_id` в этом endpoint вообще не участвует в логике
- у части веб-пользователей с ID вида `web_*` `contact_id` пустой, поэтому кнопка выглядела как нерабочая

## Что изменено

- проверка перед `fetch('/api/cancel-subscription')` переведена на `telegramId`
- success-сообщение в настройках уточнено: отменяется автосписание, но подписка остаётся активной до конца оплаченного срока

## Архитектурный вывод

- для рекуррентных платежей `telegram_id` по-прежнему является строковым ключом пользователя в БД
- `web_*` — валидный идентификатор для `create-payment`, `yookassa-webhook`, `cron-subscriptions` и `cancel-subscription`
- нельзя блокировать эти сценарии по отсутствию `contact_id`
- отдельно стоит помнить, что Telegram-уведомления для `web_*` могут не доходить, потому что это уже не chat id
