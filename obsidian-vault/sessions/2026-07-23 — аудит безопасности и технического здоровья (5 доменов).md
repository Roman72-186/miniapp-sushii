# Аудит безопасности и технического здоровья — 2026-07-23

Пять параллельных субагентов разобрали код по зонам риска. Ниже — полный список находок с указанием, что уже исправлено в этой сессии, а что осталось на потом.

## Статус исправлений

### ✅ Исправлено 2026-07-23

1. **`api/export-contacts.js`** — публичный дамп всей базы клиентов без авторизации → добавлена проверка admin-токена (`checkAuth`).
2. **`api/yookassa-webhook.js`** — подмена `X-Forwarded-For` обходила проверку подлинности вебхука; тело запроса использовалось без верификации → убрана проверка по IP, платёж теперь **всегда** верифицируется через API YooKassa по `payment.id`, вся дальнейшая логика берёт данные только из проверенного ответа.
3. **Системное отсутствие JWT-проверки** на 15 пользовательских эндпоинтах (`get-profile`, `get-order-history`, `get-transactions`, `get-gift-history`, `get-referrals`, `claim-gift`, `cancel-subscription`, `register-referral`, `apply-partner-code`, `user-order-rating`, `check-vip`, `sync-user`, условно `create-order`/`create-payment` с сохранением анонимного гостевого чекаута) → identity теперь везде берётся из `req.userId` (JWT), а не из тела запроса.

Легаси-путь идентификации по числовому `telegram_id` без JWT (Telegram WebApp + URL-параметр `?telegram_id=` в `UserContext.js`) сознательно не восстанавливался — по решению владельца эти пользователи больше не активны. Побочный эффект: `CompleteRegistrationPage.js` (флоу завершения веб-регистрации легаси-юзеров) теперь получает 401 от `get-profile` вместо профиля.

`check-vip.js` защищён JWT, но там остался отдельный не связанный баг: для веб-пользователей `telegram_id` — строка `web_xxx`, а не настоящий числовой Telegram ID, поэтому проверка членства в VIP-группе и раньше не работала для них корректно. Не трогали — отдельная задача.

### ⏳ Не исправлено — критично

- **Сломанная вкладка «Баннеры» в `AdminPage.js`** — три независимых JSX-блока подряд под одним условием `tab === 'banners'` (строки ~1242, ~1291, ~1524): старая сетка баннеров, полная детальная панель подписчиков (явно предназначенная для вкладки «Люди»), актуальный редактор баннеров. Реально показывается смесь всех трёх.
- **Утечка API-ключа TestSprite в git-историю** — `opencode.jsonc`, коммит `827715f`. Ключ нужно ротировать, историю — почистить (`git filter-repo`/BFG).
- **Двойной запуск крона подписок без блокировки** — встроенный `setTimeout`-крон в `server.js` и внешний `/api/cron-subscriptions` могут отработать параллельно. Частично смягчено уже существующим `pending: true` в `tryRecurringPayment` (см. ниже, было уже на диске до этой сессии), но полноценного mutex/lock всё ещё нет.
- **Захват аккаунта через email-OTP** — пока у пользователя не установлен пароль, `send-email-otp.js` не сверяет присланный email с уже привязанным; зная чужой телефон, можно привязать свою почту и получить JWT чужого аккаунта.

### ⏳ Не исправлено — важно

- **Гонка идемпотентности вебхука** — `yookassa_payment_id` без `UNIQUE` в схеме (ни SQLite, ни `scripts/create-pg-schema.sql`). При двойной доставке вебхука YooKassa возможна двойная запись платежа и двойное начисление SHC.
- **Перегруженное поле `tariff`** — разовая покупка амбассадорского тарифа (9990) перезаписывает `tariff` у пользователя с активной подпиской-подпиской, из-за чего автосписание спишет не ту сумму.
- **Гонка при списании SHC-баллов** — проверка баланса и списание разнесены во времени, `updateBalance` не атомарен (`UPDATE ... SET balance_shc = balance_shc + ?` без условия `>= 0`), баланс может уйти в минус при повторном запросе.
- **Таблица `orders` отсутствует в PG-схема-скриптах** (`scripts/create-pg-schema.sql`, `scripts/setup-pg-schema.js`) — при развёртывании базы с нуля сломаются email-уведомления и игра «Пять букв» (общий try/catch в `ensureMigrations()` в `db-pg.js`), а история заказов при разовой миграции (`migrate-sqlite-to-pg.js`) потеряется безвозвратно.
- **Сломанный прогресс-бар подписки в `ProfilePage.js`** — опечатка в регистре ключа (`профиль.датаНАЧАЛА` вместо `profile.датаНачала`, строки ~132-134), фича тихо не работает с момента написания.
- **Промокод не сбрасывается после заказа** в `DiscountShopPage.js` (`handleOrderSuccess`) — при следующей покупке поле остаётся заблокированным старым кодом.
- **Fallback `JWT_SECRET` захардкожен** в `api/_lib/auth.js` и `api/update-profile.js` (`'your-secret-key-change-in-production'`) — при отсутствии переменной окружения сервер тихо подписывает токены известным значением вместо падения с ошибкой на старте.
- **Оракул для перебора телефонов** в `api/auth/login-by-phone.js` — ответ отличается в зависимости от существования пользователя и наличия пароля, rate-limit отсутствует.
- Отсутствуют security-заголовки (HSTS, CSP, X-Frame-Options) и rate limiting в `nginx/default.conf` на auth/admin-эндпоинтах.
- Docker-контейнер (`Dockerfile`) запускается от root, нет `USER` директивы.
- `.dockerignore` не исключает `data/`, `*.db`, `opencode.jsonc`, `.secrets.lock.local`.
- Нет backup-стратегии для Docker volume `app-data`.

### ⏳ Технический долг — без спешки

- Дублирование логики нормализации телефона в трёх местах (`PaymentPage.js`, `ProfilePage.js`, `EditProfileModal.js`) вместо единого `src/utils/phone.js`.
- Рассинхрон написания названий товаров между каталогами (Ё/Е, Э/Е, дефисы) — ломает синхронизацию `enabled`-флага в `SYNC_GROUPS` (`api/admin-products.js`). Конкретные позиции: «Суши сэндвич/сендвич с лососем» (rolls/rolls-sub/rolls-490), «Запечённый/Запеченный Мидзуи» и ещё 3 позиции (zaproll/zaproll-sub), «Суши пицца №1-3» vs «Суши-пицца лосось/креветки/курица».
- Числовые PG-колонки возвращаются как строки без явного каста (`getMonthRevenue`, `getTotalEarnings` в `db-pg.js`) — выручка в админке отображается без разделителя разрядов из-за смены бэкенда (SQLite → PG).
- PII (телефон, адрес) в открытом виде в логах `create-order.js`.
- Неатомарная запись каталогов в `data/products/*.json` (`api/admin-products.js`, `api/_lib/gift-rules.js`) — в отличие от уже правильно сделанных `user-cache.js`/`blob-store.js` (tmp+rename).
- Отладочные `console.log` в горячих путях (`subscription-state.js`, `gift-windows.js`, `DiscountShopPage.js`, `sync-user.js`).
- Дублирование `formatDate` в `yookassa-webhook.js` и `cron-subscriptions.js`.
- Мёртвый экспорт `processCommissions` в `db.js`/`db-pg.js` (амбассадорские комиссии намеренно отключены).
- Refresh-токен (`generateRefreshToken` в `auth.js`) не даёт независимой защиты (тот же `JWT_SECRET + '_refresh'`) и нигде не используется бэкендом.
- Отсутствие `UNIQUE` на `phone` в SQLite (`users` таблица) → race condition при параллельной регистрации на один номер.

## Полные отчёты субагентов (для справки)

### 1. Платежи и подписки (денежный контур)

Файлы: `api/create-order.js`, `api/yookassa-webhook.js`, `api/cancel-subscription.js`, `api/cron-subscriptions.js`, `api/_lib/subscription-state.js`, `api/_lib/gift-windows.js`, `api/_lib/db.js`/`db-pg.js`.

**Критично:**
1. Подмена `X-Forwarded-For` обходила проверку подлинности вебхука (см. «Исправлено» выше).
2. Даже при успешной верификации через API использовались недоверенные поля из тела запроса (см. «Исправлено» выше).

**Важно:**
3. Гонка идемпотентности вебхука — нет `UNIQUE` на `yookassa_payment_id`.
4. Перегруженное поле `tariff` между «текущий тариф подписки» и «разовая покупка амбассадора» → неверная сумма автосписания.
5. Гонка состояний в списании SHC-баллов — нет атомарности, нет защиты от отрицательного баланса.

**Рекомендации:**
6. Недостижимый код в `cron-subscriptions.js` (`tryRecurringPayment` всегда возвращает `pending: true`, ветка `success && !pending` мертва — на момент аудита это уже было на диске несохранённым изменением).
7. Расхождение между заявленным планом уведомлений (3 дня) и реализацией (только 1 день).
8. Избыточные debug `console.log` в горячих путях.
9. PII в логах `create-order.js`.
10. Дублирование `formatDate`, мёртвый экспорт `processCommissions`.

Положительно: SQL-инъекций не найдено, округление копеек корректно, бизнес-логика идентична между SQLite и PostgreSQL.

**Рекомендации:**
