### 2. Ввод «Ё» через long-press на «Е» (iOS-style)
- `src/components/WordleKey.js` переписан:
  - Для клавиш **без `altChars`** — поведение не изменилось (обычный `onClick`).
  - Для клавиш **с `altChars`** — pointer events (`pointerdown`/`pointermove`/`pointerup`/`pointercancel`), `setPointerCapture` на кнопке, таймер 450ms → popup. Выбранный элемент определяется по `getBoundingClientRect` в `pointermove`; финальный выбор — по координате `pointerup`.
  - Короткий тап (< 450ms) → обычный ввод `char`.
  - Отпускание вне popup'а — отмена без ввода.
- `src/components/WordleKeyPopup.js` **не создавал** — popup рендерится внутри самой кнопки (проще управлять состоянием + позиционированием).
- CSS popup'а (`.wrd-key-popup`, `.wrd-key-popup__item`, `.wrd-key-popup__item--active`): закруглённый прямоугольник над кнопкой с «носиком» вниз (`::after` triangle), анимация входа 120ms, активный пункт подсвечивается зелёным.
- Индикатор альтернатив на кнопке «е» — маленькая зелёная точка в правом углу (`.wrd-key--has-alt::after`).
- `src/WordlePage.js`: передаётся `altChars={key === 'е' ? ['ё'] : undefined}`, контракт `onClick` изменён на `(char) => void`.

## Решения
- **Popup внутри кнопки, не отдельным компонентом**: проще с состоянием, проще `position: absolute` относительно кнопки, меньше моста между родителем и детьми.
- **Pointer Events вместо touch+mouse**: один API покрывает тач, мышь, стилус. Desktop-разработка тестится long-click'ом ЛКМ.
- **setPointerCapture + elementFromPoint-подобная логика** — стандартный способ для контроля финального выбора при drag-to-select. Без capture'а тач-события теряются после ухода пальца с изначального target'а.
- **Не делал отдельный ряд с Ё**: user явно попросил iOS-подобную клавиатуру (long-press), а не видимую кнопку.
- **Индикатор-точка на «е»** — UX-подсказка, что есть альтернатива.
