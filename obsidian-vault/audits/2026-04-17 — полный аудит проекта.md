## Executive Summary

**Общее состояние:** Проект функционально стабилен, активно развивается, имеет хорошо документированную архитектуру и чёткое разделение ответственности. Основные бизнес-процессы (платежи, подписки, заказы) работают корректно. Есть серьёзные технические риски, связанные с безопасностью, устойчивостью кэшей и отсутствием автобэкапов.

**Главные победы:**
1. Грамотная двухуровневая архитектура данных — SQLite/PostgreSQL proxy с одинаковыми API, готовность к масштабированию
2. Комплексная система кэширования — файловый кэш (TTL 5 мин), blob-store, HTTP-кэш статики
3. Безопасность платежей — верификация YooKassa webhook (IP + fallback API), идемпотентность
4. Детальная документация — обширный CLAUDE.md, Obsidian knowledge base
5. Современный стек — React 19, Node 20, JWT-auth, Docker с nginx + SSL

**Главные риски:**
1. **CRITICAL**: Админский пароль по умолчанию `test123` в production коде (`api/_lib/admin-auth.js:10`)
2. **HIGH**: Риски SQL injection через динамические запросы в admin endpoints
3. **HIGH**: Файловые кэши без блокировок — race conditions при concurrent записи
4. **MEDIUM**: Отсутствие мониторинга, structured logging, автобэкапов БД
5. **MEDIUM**: Legacy Telegram код увеличивает attack surface

## Детальный разбор

### 1. Архитектура и структура кода

**Фронтенд (`src/`)**
- ✅ Чёткое разделение: компоненты, хуки, утилиты, страницы
- ✅ Neumorphic дизайн-система через `shop-v2.css` (830 строк CSS-override)
- ⚠️ Routing через `window.location.pathname` (App.js:27-29) — работает, react-router был бы надёжнее
- ❌ Legacy Telegram код в `UserContext.js:44-65` (WebApp SDK, `initDataUnsafe`) — помечен «dead weight», но читается

**Бэкенд (`api/`, `server.js`)**
- ✅ Модульная структура, `_lib/` для shared utilities
- ✅ Корректный CORS на всех endpoints
- ⚠️ Архитектурная асимметрия: `app.all()` vs `app.post()`

**Dev/prod, SQLite↔PostgreSQL**
- ✅ Proxy паттерн в `db.js:2-5` — переключение по `USE_SUPABASE`
- ✅ Одинаковые API между `db.js` и `db-pg.js`
- ⚠️ Runtime-миграции разные для SQLite (try/catch) vs PostgreSQL (`IF NOT EXISTS`) — потенциал рассинхронизации схем

### 2. Безопасность

**JWT auth**
- ✅ JWT с 7-дневным TTL, корректная верификация в middleware
- ✅ Supabase для паролей — разделение credentials и data
- ❌ **CRITICAL**: `api/_lib/admin-auth.js:10` — `process.env.ADMIN_PASSWORD || 'test123'`

**Admin auth**
- ✅ Токены в памяти с TTL 24ч, не persisted
- ❌ Нет строгой валидации параметров admin endpoints

**Валидация ввода**
- ⚠️ Админские endpoints используют динамическое построение запросов — нужен аудит prepared statements
- ✅ React экранирует по умолчанию, нет `dangerouslySetInnerHTML`
- ✅ Редиректы только внутренние

**YooKassa webhook**
- ✅ Двойная верификация: IP + API fallback (`yookassa-webhook.js:31-42`)
- ✅ Идемпотентность: проверка дублей по `yookassa_payment_id`

### 3. Данные и миграции

**Runtime-миграции (`ALTER TABLE ADD COLUMN`)**
- ⚠️ Разные стратегии SQLite vs PG — ручная синхронизация схем
- ⚠️ Нет версионирования миграций — не ясно, что применено на каком инстансе

**Файловые кэши**
- ❌ **Race conditions**: `user-cache.js:30-34`, `blob-store.js` — concurrent writes без file locking
- ✅ TTL 5 минут корректен (`sync-user.js:9`)
- ⚠️ Некоторые admin endpoints не вызывают `deleteUserCache` после update

**Консистентность**
- ✅ Синхронизация `users.name` ↔ `first_name/last_name` в `update-profile.js`
- ⚠️ `phone normalization` — 4+ места с дублированной логикой

### 4. Платежи и бизнес-логика

- ✅ Идемпотентность webhook, проверка дублей
- ✅ SHC начисления: `processReferralBonus`, `processReferralSHC` активны
- ✅ Амбассадорские комиссии намеренно отключены (документировано в CLAUDE.md)
- ⚠️ `claim-gift.js` — claim без atomic lock: теоретическая возможность double-claim
- ✅ `buildWindows` (`api/_lib/gift-windows.js`) — математика корректна

### 5. Каталоги и админка

- ⚠️ Приоритет `data/products/` над `build/` — известная ловушка: новые товары в subscription caталогах могут не появиться, пока не обновлены overrides на VPS
- ✅ Sync groups работают (`rolls ↔ rolls-sub ↔ rolls-490` и т.д.)
- ❌ Нет автоматизации чистки overrides при глобальных изменениях каталога
- ⚠️ Новые изображения требуют ручного `git add` — deploy ломается если забыли

### 6. Производительность и надёжность

- ❌ **N+1** в AdminPage subscribers tab — batch query решил бы
- ⚠️ Словарь игры (6074 слова) грузится в память при старте — приемлемо, но мог бы быть lazy
- ❌ **No file locking** в кэшах — corrupted JSON возможен под concurrency
- ⚠️ React bundle не оптимизирован (нет lazy routes, tree shaking spot-check)
- ❌ `setTimeout` cron в `server.js` — при рестарте контейнера пропадает текущий таймер, но следующий запуск запланирует заново; риск — пропуск дня, если рестарт попадёт в окно

### 7. Деплой и инфраструктура

- ✅ Docker multi-stage, persistent volume `app-data`
- ✅ Nginx + Let's Encrypt
- ❌ Нет автобэкапов БД (`npm run backup:db` есть, но не в cron на VPS)
- ❌ Нет health check endpoint, внешнего мониторинга, alerts
- ❌ Логи — `console.log`, не структурированы

### 8. Тесты, линтинг, DX

- ❌ Тестов практически нет (только CRA default `App.test.js`)
- ❌ Нет ESLint/Prettier конфига
- ❌ JavaScript без TypeScript — нет compile-time проверок
- ⚠️ Дубли: phone normalization, imageMap ключи

### 9. Технический долг

- ❌ Legacy Telegram код в `UserContext.js`, `sync-user.js`, query-param `?telegram_id=` — attack surface + когнитивная нагрузка
- ⚠️ `users.name` пишется параллельно с `first_name/last_name` — планово, но стоит план удаления
- ⚠️ Phone normalization дублируется в 4 местах

## Что делать НЕ нужно

Чтобы не уйти в overengineering:

1. **НЕ переписывать на TypeScript** — проект стабилен, migration effort непропорционален текущему масштабу
2. **НЕ внедрять Redux/Zustand** — UserContext + localStorage достаточны
3. **НЕ переходить на react-router** — ручной routing работает, замена не даст новой ценности
4. **НЕ переносить кэши на Redis** — файловой системы хватает под текущую нагрузку
5. **НЕ делать строгую схему-миграцию (Prisma/Drizzle/Alembic)** — новых полей мало, runtime ALTER работает
6. **НЕ оптимизировать N+1 в админке пока юзеров <1000** — не блокирует, важнее security и backups
